# 운영자 매뉴얼

## 1. 목적과 범위

이 문서는 동국제강 원료기획팀 실적 모니터링 대시보드의 운영 담당자를 위한 매뉴얼입니다. 계정 관리, 공지/일정, 거래처, 원본 실적 데이터, 계획 데이터, 등급 매핑, 보고서 내보내기 등 시스템 전반의 운영 절차와 주의사항을 포괄합니다.

---

## 2. 시스템 개요

본 시스템은 별도 백엔드 서버 없이 동작하는 정적 웹 대시보드입니다. 모든 데이터는 브라우저 내장 저장소(IndexedDB/localStorage)에 저장됩니다.

### 2.1 주요 파일 구성

| 구성 요소 | 역할 | 크기/특성 |
| --- | --- | --- |
| `login.html` | 로그인 화면 | 인증, 자동 리다이렉트, 테스트 계정 |
| `index.html` | 메인 대시보드 | 탭 구성, 모달 7종, 챗봇 UI |
| `js/dashboard-data.js` | 기본 데이터 번들 | Python 스크립트로 생성 |
| `js/storage.js` | 저장소 추상화 | IndexedDB + localStorage 하이브리드 |
| `js/admin-features.js` | 관리 기능 | 공지사항, 일정, 사용자 CRUD |
| `js/app.js` | 메인 앱 로직 | ~4,300줄 IIFE, 차트/표/챗봇/내보내기 |
| `scripts/build_dashboard_data.py` | 데이터 생성 | XLSX → JS 데이터 번들 변환 |

### 2.2 스크립트 로드 순서

`index.html` 하단에서 다음 순서로 로드됩니다. 순서가 바뀌면 동작하지 않습니다.

1. `js/storage.js` → `window.appStorage` 정의
2. `js/dashboard-data.js` → `window.dashboardData` 정의
3. `js/admin-features.js` → 공지/일정/사용자 관리 함수 전역 노출
4. `js/app.js` → IIFE 실행, `appStorage.ready` 후 초기화

### 2.3 외부 의존성

| 라이브러리 | 버전 | 로드 방식 | 용도 |
| --- | --- | --- | --- |
| Chart.js | 4.4.1 | CDN (`<head>`) | 차트 렌더링 |
| docx | 8.5.0 | CDN (`<body>` 하단) | DOCX 보고서 생성 |

## 3. 실행과 접속

### 3.1 로컬 실행

```bash
python -m http.server 8787 --bind 127.0.0.1
```

브라우저에서 `http://127.0.0.1:8787/login.html`에 접속합니다.

### 3.2 기본 계정

| 아이디 | 비밀번호 | 이름 | 부서 | 직급 | 권한 | 상태 |
| --- | --- | --- | --- | --- | --- | --- |
| `dongkuk1` | `1234` | 이돈석 | 원료기획팀 | 팀장 | 관리자 | 활성 |
| `dongkuk2` | `1234` | 박영수 | 원료기획팀 | 대리 | 일반 사용자 | 활성 |

### 3.3 로그인 흐름

1. `login.html` 접속 시 이미 `sessionStorage.loggedInUser`가 있으면 `index.html`로 자동 리다이렉트됩니다.
2. 사용자 조회 우선순위: `appStorage(IndexedDB)` → `localStorage` → 하드코딩 기본 계정 2건.
3. 비활성(`inactive`) 계정은 "비활성화된 계정입니다. 관리자에게 문의하세요." 메시지와 함께 로그인이 차단됩니다.
4. 로그인 성공 시 `sessionStorage`에 `{id, name, dept, role}` JSON 객체가 저장됩니다.
5. `index.html`의 `<head>` 인라인 스크립트가 `sessionStorage.loggedInUser` 부재 시 `login.html`로 리다이렉트합니다.

### 3.4 로그아웃

`sessionStorage`에서 `loggedInUser`를 제거하고 `login.html`로 이동합니다.

## 4. 저장 구조와 운영 특성

### 4.1 저장소 계층

| 계층 | 저장소 | 용도 |
| --- | --- | --- |
| 기본 | IndexedDB `dongkuk_dashboard` / 스토어 `kv` | 운영 데이터 영속 저장 |
| 대체 | localStorage | IndexedDB 사용 불가 시 자동 전환 |
| 세션 | sessionStorage | 로그인 세션 유지 (`loggedInUser`) |

### 4.2 저장소 API (`window.appStorage`)

| API | 설명 |
| --- | --- |
| `appStorage.ready` | DB 초기화 + 마이그레이션 완료 Promise |
| `appStorage.getSync(key)` | 인메모리 캐시에서 동기 읽기 |
| `appStorage.get(key)` | Promise 기반 비동기 읽기 |
| `appStorage.set(key, val)` | 캐시 먼저 갱신 → 비동기 IDB 쓰기 |
| `appStorage.remove(key)` | 캐시 + IDB에서 삭제 |

### 4.3 주요 저장 키

| 키 | 내용 | 데이터 형식 |
| --- | --- | --- |
| `planClipboardDataByYear` | 연도별 계획/실적 붙여넣기 데이터 | `{data, timestamp}` 봉투 |
| `rawTransactionDataByYear` | 연도별 원본 실적 데이터 | `{data, timestamp}` 봉투 |
| `gradeMacroMappings` | 검수 등급 → 대분류 매핑 | `{data, timestamp}` 봉투 |
| `supplierAdminItems` | 거래처 관리 데이터 | `{data, timestamp}` 봉투 |
| `noticesData` | 공지사항 (사용자 생성분만) | `{data, timestamp}` 봉투 |
| `schedulesData` | 팀 일정 데이터 | `{data, timestamp}` 봉투 |
| `usersData` | 사용자 데이터 | `{data, timestamp}` 봉투 |

### 4.4 저장소 마이그레이션

최초 실행 시 `localStorage`에 기존 데이터가 있으면 IndexedDB로 자동 마이그레이션합니다.

- `localStorage.__idb_migrated` 플래그로 중복 마이그레이션을 방지합니다.
- 위 7개 키를 순회하며 JSON 파싱 → IDB 쓰기 → localStorage 삭제합니다.
- 개별 키 마이그레이션 실패는 무시하고 나머지를 계속 처리합니다.

### 4.5 관리 기능 저장 패턴

`admin-features.js`는 통일된 패턴을 사용합니다:

- **읽기** (`readStoredData`): `appStorage.getSync()` → `localStorage` 순서로 조회. `{data: [...], timestamp: "..."}` 봉투 구조를 기대합니다. 파싱 실패 시 해당 키를 삭제하고 `null`을 반환합니다.
- **쓰기** (`writeStoredData`): `{data, timestamp}` 봉투로 감싼 뒤 저장합니다.

### 4.6 운영 시 반드시 알아둘 점

- 브라우저 로컬 저장 방식이므로 PC나 브라우저 프로필이 다르면 데이터가 공유되지 않습니다.
- 브라우저의 "사이트 데이터 삭제" 실행 시 운영 데이터가 모두 사라집니다.
- 서버 동기화, 중앙 백업, 권한 분리 기능은 구현되어 있지 않습니다.
- 비밀번호는 평문으로 브라우저 저장소에 보관됩니다 (암호화 없음).

## 5. 캐싱 시스템

`app.js`는 성능을 위해 2단계 캐시를 운영합니다.

### 5.1 1단계: 트랜잭션 캐시 (`_txCache`)

- `getRawTransactionsForYear()` 결과를 캐싱합니다.
- 캐시 키: `"${year}:${rowCount}:${_gradeMappingsVersion}"`
- 대분류(macro) 필드를 원본 행에 직접 적용합니다 (복제하지 않음, 성능 최적화).

### 5.2 2단계: 집계 캐시 (`_aggCache`)

| 슬롯 | 대상 함수 | 캐시 키 |
| --- | --- | --- |
| `suppliers` | `buildSupplierDatasetFromTransactions()` | 트랜잭션 캐시 키와 동일 |
| `purchases` | `buildPurchasesDatasetFromTransactions()` | 트랜잭션 캐시 키와 동일 |
| `gradeImport` | `buildGradeImportDatasetFromTransactions()` | `"${txCacheKey}\|${compareTxCacheKey}"` |

### 5.3 캐시 무효화

`_invalidateTxCache()` 호출 시 양쪽 캐시가 모두 삭제됩니다. 다음 상황에서 호출됩니다:

- 연도 변경 시
- 원본 실적 데이터 초기화 시
- 검수 등급 매핑 변경 시 (`_gradeMappingsVersion` 카운터 증가)

## 6. 계정과 권한 운영

### 6.1 사용자 등록

1. `사용자관리` 탭으로 이동합니다.
2. `+ 사용자 등록`을 클릭합니다.
3. 다음 항목을 입력합니다:

| 항목 | 규칙 |
| --- | --- |
| 아이디 | 영문, 숫자, 밑줄(`_`)만 허용. 중복 불가 |
| 비밀번호 | 신규 등록 시 필수 |
| 이름 | 필수 |
| 부서 | 필수. 선택 가능 부서: 원료기획팀, 원료구매팀, 생산관리팀, 품질관리팀, 경영기획팀, 재무팀, IT팀 |
| 직급 | 선택 가능 직급: 사원, 대리, 과장, 차장, 부장, 팀장, 이사 |
| 권한 | `관리자` 또는 `일반 사용자` |
| 상태 | `활성` 또는 `비활성` |
| 이메일 | 선택. 입력 시 형식 검증 (`user@domain.tld`) |

4. `저장`을 누릅니다.

### 6.2 사용자 수정

- 아이디는 변경할 수 없습니다 (읽기 전용).
- 비밀번호를 비워 두면 기존 비밀번호가 유지됩니다.
- 상태를 `비활성`으로 변경하면 해당 계정은 즉시 로그인 불가 상태가 됩니다.
- 사용자 정보를 수정하면 현재 로그인된 사용자의 헤더 표시(이름/부서)도 자동 갱신됩니다 (`window.refreshLoggedInUserDisplay` 호출).

### 6.3 사용자 삭제

- 현재 로그인한 계정은 삭제할 수 없습니다 (`sessionStorage`의 로그인 ID와 비교).
- 삭제 전 확인창(`window.confirm`)이 표시됩니다.

### 6.4 기본 사용자 처리

- 시스템 기본 사용자 2건(`dongkuk1`, `dongkuk2`)은 저장소에 사용자 데이터가 없을 때 폴백으로 사용됩니다.
- 사용자를 등록/수정하면 저장소에 전체 사용자 목록이 저장되고, 이후로는 저장소 데이터가 우선합니다.

### 6.5 사용자 목록 정렬 및 페이징

- 관리자 계정이 상단에 표시됩니다.
- 이후 생성일 역순으로 정렬됩니다.
- 페이지당 10건 표시됩니다.

### 6.6 KPI 표시

| KPI | 설명 |
| --- | --- |
| 전체 사용자 | 등록된 사용자 총 수 |
| 활성 사용자 | `status === "active"` 수 |
| 관리자 | `role === "admin"` 수 |
| 비활성 사용자 | `status === "inactive"` 수 |

### 6.7 권한 운영 주의

현재 `role` 값은 사용자 정보 표시와 목록 구분에 사용되지만, **메뉴 접근 자체를 제한하지는 않습니다**. 로그인한 사용자는 모든 탭을 볼 수 있습니다. 운영 시에는 계정 발급 정책으로 접근 대상을 통제해야 합니다.

## 7. 정기 운영 절차

### 7.1 월별 계획/실적 데이터 갱신

**대상 화면**: `부재료실적 모니터링 > 월별 계획 대비 실적 데이터` (접힘 패널)

**입력 방법 A: 엑셀에서 복사/붙여넣기**

1. 접힘 패널을 펼칩니다 (토글 클릭).
2. 엑셀에서 아래 형식의 데이터를 복사합니다:

```
        1월    2월    3월   ...  12월
인천계획  xxx   xxx   xxx   ...   xxx
인천실적  xxx   xxx   xxx   ...   xxx
포항계획  xxx   xxx   xxx   ...   xxx
포항실적  xxx   xxx   xxx   ...   xxx
```

3. 그리드에 붙여넣습니다 (paste 이벤트 자동 인식).

**스마트 파싱 로직**:
- 먼저 월 헤더 행을 탐색합니다 (`extractMonthColumns`). 1~12월 중 6개월 이상이 연속으로 있어야 인식합니다.
- `인천`/`포항` 키워드와 `계획`/`실적` 키워드로 행을 자동 식별합니다 (`identifyPlanPasteRows`).
- 키워드 식별에 실패하면, 숫자값 12개 이상인 행이 정확히 4개일 경우 순서대로 인천계획/인천실적/포항계획/포항실적으로 해석합니다 (폴백).
- 파싱 실패 시 한국어 오류 메시지가 표시됩니다.

**입력 방법 B: 그리드 직접 편집**

1. 그리드의 각 셀을 직접 클릭하여 값을 입력합니다.
2. 셀 포커스 시 기존 값이 전체 선택됩니다.
3. Enter 키를 누르면 다음 셀로 이동합니다.
4. 포커스를 벗어나면 숫자 포맷(`toLocaleString`)이 자동 적용됩니다.
5. `붙여넣은 값 적용` 버튼을 클릭하여 데이터를 반영합니다.

**복원 방법**:
- `기본 데이터 복원`을 누르면 해당 연도의 기본 수급계획 데이터(`dashboardData`)로 되돌립니다.
- 사용자가 입력한 데이터는 연도별로 `planClipboardDataByYear` 키에 저장됩니다.

### 7.2 원본 실적 데이터 갱신

**대상 화면**: `부재료실적 모니터링 > 거래처 납품 실적관리 데이터 (인천공장)` (접힘 패널)

**입력 절차**:

1. 상단에서 대상 연도를 먼저 선택합니다.
2. 접힘 패널을 펼칩니다.
3. 엑셀에서 A:E 열을 다음 순서로 복사합니다:

| 열 | 항목 | 형식 | 예시 |
| --- | --- | --- | --- |
| 1 | 전기일 | `YYYY-MM-DD` | 2024-03-15 |
| 2 | 공급업체 | 텍스트 | 현대스크랩 |
| 3 | 검수등급 | 텍스트 | 국고 하 |
| 4 | 단가 | 숫자 | 250000 |
| 5 | 입고금액 | 숫자 | 75000000 |

4. 그리드에 붙여넣습니다.
5. `원본 데이터 반영`을 클릭합니다.
6. 거래처 추이, 구매실적, 등급별현황/수입관리 화면이 정상 갱신되었는지 확인합니다.

**수량 계산 방식**:
- 수량은 `입고금액 / 단가`로 재계산됩니다: `qty = unitPrice ? (amount / unitPrice + 0.5) | 0 : 0`
- 화면 그리드의 4열 제목은 `입고량`으로 보이지만, 실제 파서는 4열을 `단가`로 해석합니다.
- **반드시 `단가` 값을 붙여넣어야 합니다.**

**대용량 처리**:
- 2만 건 이하: 동기 처리 (`parseRawTransactionText`).
- 2만 건 초과: 비동기 청크 처리 (`parseRawTransactionTextAsync`, 2만 행/청크, `setTimeout(0)` 기반). 진행률 표시가 나타납니다.

**그리드 페이징**:
- 페이지당 100건 표시됩니다.
- 최소 18행이 항상 표시됩니다 (빈 행 자동 추가).
- 처음/이전/번호/다음/마지막 페이지 이동 버튼이 있습니다.

**초기화**:
- `선택 연도 초기화`를 누르면 해당 연도의 원본 실적 데이터를 제거합니다.
- 초기화 시 캐시가 즉시 무효화됩니다.

**반영 범위**:

| 화면 | 반영 여부 |
| --- | --- |
| 거래처 추이 차트 (상위 3곳) | O (재계산) |
| 구매실적 탭 전체 | O (재계산) |
| 등급별현황/수입관리 탭 | O (재계산) |
| 인천공장 등급 도넛 차트 | O (재계산) |
| 거래처 관리 테이블/점유율 차트 | X (기존 번들 데이터 유지) |

### 7.3 검수 등급 매핑 관리

**대상 화면**: `부재료실적 모니터링 > 검수 등급 매핑 관리` (접힘 패널)

**대분류 5종** (고정):

| 대분류 | 설명 |
| --- | --- |
| 국고상 | 상급 국내 고철 |
| 국고중 | 중급 국내 고철 |
| 국고하 | 하급 국내 고철 |
| 선반설 | 선반/설비 고철 |
| 기타 | 미분류 |

**등록 절차**:
1. 대분류 드롭다운에서 상위 분류를 선택합니다.
2. 상세 검수등급명을 텍스트 입력합니다.
3. `매핑 추가`를 누릅니다.
4. 추가된 매핑은 한국어 알파벳 순으로 자동 정렬됩니다.

**삭제**: 각 매핑 칩의 `삭제` 버튼으로 개별 삭제합니다.

**기본값 복원**: `기본값 복원` 클릭 시 `window.confirm()` 확인 후 약 30개의 기본 매핑으로 복원됩니다.

**영향 범위**: 매핑 변경 시 `_gradeMappingsVersion` 카운터가 증가하여 모든 캐시가 무효화되고, 다음 렌더링 시 등급별 현황/배분/비교 표가 새 매핑 기준으로 재계산됩니다.

**매핑 적용 방식**: 원본 실적 데이터의 상세등급명이 매핑에 없으면 자동으로 `기타`로 분류됩니다. 등급명 비교 시 공백을 포함하여 정규식으로 처리합니다 (예: `국고 하` → `국고하`).

### 7.4 거래처 관리

**대상 화면**: `거래처 관리` 탭

**등록 항목**:

| 항목 | 필수 | 설명 |
| --- | --- | --- |
| 코드 | O | 자동 제안 (`S{max+1}`, 3자리 제로패딩). 신규 등록 시 자동 입력 |
| 거래처명 | O | 텍스트 |
| 지역 | O | 텍스트 |
| 대표자 | O | 텍스트 |
| 연락처 | O | 텍스트 |
| 이메일 | - | 형식 검증 |
| 신뢰등급 | - | A / A- / B+ / B |
| 월 공급능력(톤) | - | 숫자 |
| 금년 납품량(톤) | - | 숫자 |

**성과율 자동 계산**: `performanceRate = clamp(round((yearlySupply / monthlyCapacity) × 12), 1, 99)`. 사용자가 직접 입력하지 않아도 월 공급능력과 금년 납품량으로 자동 산출됩니다.

**성과율 색상 표시**:

| 범위 | 색상 | CSS 클래스 |
| --- | --- | --- |
| 90% 이상 | 녹색 | `.good` |
| 85% 이상 | 주황 | `.mid` |
| 85% 미만 | 빨강 | `.low` |

**신뢰등급 배지 색상**: A→녹색, A-→파랑, B+→주황, 기타→회색.

**수정 모드**: 코드가 읽기 전용입니다.

**삭제**: `window.confirm()`으로 거래처명을 포함한 확인 메시지가 표시됩니다.

**기본 거래처**: 8건(S001~S008)이 하드코딩되어 있으며, 저장소에 데이터가 없을 때 사용됩니다.

**KPI 표시**:

| KPI | 계산 |
| --- | --- |
| 전체 거래처 | 등록 수 |
| 월 공급능력 합계 | `sum(monthlyCapacity)` |
| 연간 납품량 합계 | `sum(yearlySupply)` |
| 평균 성과율 | `avg(performanceRate)` |

### 7.5 공지사항 운영

**대상 화면**: `공지사항` 탭

**기본 공지 5건**: ID `N-default-001` ~ `N-default-005`. 시스템 안내, 3월 정기회의, 포항 재고 결과, 일본 HMS1 시황, 원료 조달 전략이 포함됩니다. 이 중 001, 002번은 필독(핀) 설정됩니다.

**기본 공지 저장 방식**:
- 기본 공지는 저장소에 저장되지 않습니다.
- 매번 앱 로드 시 하드코딩된 기본 공지를 먼저 로드한 후, 저장소의 사용자 작성 공지를 추가합니다.
- 기본 공지를 화면에서 삭제해도 새로고침 시 다시 나타납니다.
- 지속적으로 유지하려는 공지는 신규 공지로 별도 등록하는 것이 안전합니다.

**등록 절차**:
1. `+ 등록` 클릭
2. 제목, 내용, 작성자, 비밀번호 입력
3. 필요 시 `필독` 체크 (상단 고정 + 필독 배지)
4. `저장`

**수정/삭제 규칙**:
- 수정과 삭제 모두 게시글 등록 시 설정한 비밀번호가 필요합니다.
- 비밀번호가 일치하지 않으면 수정/삭제할 수 없습니다.
- 삭제 시 별도 비밀번호 확인 모달(`#noticeDeleteModal`)이 열립니다.

**정렬 및 페이징**:
- 필독(핀) 공지가 상단에 항상 표시됩니다 (페이징 대상 아님).
- 일반 공지는 생성일 역순으로 정렬됩니다.
- 페이지당 10건 표시됩니다.

## 8. 데이터 번들 재생성

### 8.1 실행 방법

```bash
python3 scripts/build_dashboard_data.py --source path/to/file.xlsx --output js/dashboard-data.js
```

### 8.2 스크립트 동작 원리

- Python 3 표준 라이브러리만 사용합니다 (pandas/openpyxl 불필요).
- XLSX 파일을 ZIP으로 취급하여 내부 XML을 직접 파싱합니다.
- 공유 문자열(`xl/sharedStrings.xml`)을 먼저 읽고, 셀 참조를 해소합니다.

### 8.3 시트 매핑

| 시트 | 역할 | 세부 |
| --- | --- | --- |
| `sheet3.xml` | 계획 데이터 | 행 2~7: 인천 등급별 물량, 행 7~8: 인천 계획/실적, 행 8~13: 포항 등급별 물량, 행 12~13: 포항 계획/실적 |
| `sheet4.xml` | 2024년 원본 거래 | 열 1: 엑셀 시리얼 날짜, 열 2: 공급업체, 열 3: 등급, 열 4: 수량, 열 5: 금액 |
| `sheet5.xml` | 2023년 원본 거래 | 동일 구조 |

### 8.4 월별 값 추출

`series_from_row(row)` 함수는 열 6~17 (1-based)에서 12개 월별 값을 추출합니다.

### 8.5 엑셀 날짜 변환

`excel_serial_to_date(value)`: `datetime(1899, 12, 30) + timedelta(days=float(value))`

### 8.6 출력 형식

```javascript
window.dashboardData = {
  meta: { sourceFile, generatedAt, displayNote, defaultYear, availableYears, labels },
  years: {
    "2024": { overview, plan, suppliers, purchases, allocation, gradeImport },
    "2023": { overview, plan: null, suppliers, purchases, allocation: null, gradeImport }
  }
};
```

- 2023년은 계획 데이터(`plan`)와 배분 데이터(`allocation`)가 없습니다.
- UTF-8, JSON `ensure_ascii=False`.

### 8.7 레거시 데이터 호환

`app.js`의 `normalizeDashboardData()`는 두 가지 데이터 형식을 처리합니다:
- **신규 형식** (`source.years` 존재): 그대로 사용, `availableYears`를 2023~2030으로 재설정.
- **레거시 형식** (플랫 구조): `inferLegacyPrimaryYear()`, `inferLegacyCompareYear()`로 연도를 자동 감지하고 `years` 객체를 재구성합니다.

## 10. 챗봇 운영

### 10.1 개요

우측 하단 FAB(플로팅 액션 버튼)으로 챗봇을 열 수 있습니다. DK 로봇 SVG 아이콘이 표시됩니다.

| 구매실적 | 구매, 매입, 입고금액 | `buildVerifiedPurchaseAnswer()` |
| 등급 | 등급, 비중, 국고상/중/하, 선반설 | `buildVerifiedGradeAnswer()` |
| 수입 | 수입, 선적, 도착, CFR | `buildVerifiedImportAnswer()` |
| 거래처 | 거래처, 납품, 공급사 | `buildVerifiedSupplierAnswer()` |
| 공장배분 | 배분, 공장 | `buildVerifiedAllocationAnswer()` |
| 전체 현황 | 현황, 요약, 대시보드 | `buildVerifiedOverviewAnswer()` |

### 10.3 기간 필터 지원

- 단월: `7월`, `3월`
- 범위: `3~6월`, `3월부터6월`
- 분기: `1분기`, `2분기`, `3분기`, `4분기`
- 반기: `상반기`, `하반기`
- 전체: `전체`, `연간`, `올해`, `합계`, `총계`

### 10.4 거래처 이름 인식

질의 텍스트에서 등록된 거래처명(대시보드 번들 + 거래처 관리 데이터)의 부분 일치를 탐지합니다.

### 10.5 공장 인식

`인천` 또는 `포항` 키워드를 감지하여 공장별 필터링을 수행합니다.

### 10.6 응답 검증

생성된 응답에 `undefined`, `null`, `NaN` 문자열이 포함되면 해당 의도를 건너뛰고 다음 의도를 시도합니다.

### 10.7 추천 질문 (6개)

`7월 계획과 실적은?`, `1분기 달성률`, `상반기 구매 현황`, `등급별 비중`, `거래처별 실적`, `전체 현황 요약`

## 11. 백업과 복구 가이드

### 11.1 백업 권장 방식

| 방법 | 설명 |
| --- | --- |
| 원본 엑셀 보관 | 데이터 번들 재생성의 원천 |

| 브라우저/프로필 고정 | 동일한 Chrome/Edge 프로필에서만 데이터 유지 |

### 11.2 장애 시 확인 순서

1. 브라우저를 새로고침합니다 (Ctrl+Shift+R 강력 새로고침 권장).
2. 조회 연도와 로그인 계정을 확인합니다.
3. DevTools(F12) > Application 탭에서 IndexedDB `dongkuk_dashboard` > `kv` 데이터 유무를 확인합니다.
4. 마이그레이션 여부 확인: `localStorage.__idb_migrated` 값이 `"1"`인지 확인합니다.
5. IndexedDB 데이터가 없고 localStorage에 데이터가 있으면, `localStorage.__idb_migrated`를 삭제한 뒤 새로고침하면 재마이그레이션됩니다.
6. 데이터가 모두 없으면 원본 엑셀에서 데이터를 다시 붙여넣습니다.

### 11.3 데이터 초기화

다음 경우 데이터가 초기화됩니다:

- 브라우저 사이트 데이터 삭제
- 시크릿/InPrivate 모드 사용 (세션 종료 시 삭제)

## 12. 운영 유의사항

### 12.1 보안 관련

- 계정 비밀번호와 공지 비밀번호는 평문으로 브라우저 저장소에 보관됩니다.
- 서버 측 인증과 암호화 저장은 구현되어 있지 않습니다.
- 정식 운영 환경으로 확대하려면 서버 DB, 중앙 인증, 권한 제어를 별도로 설계해야 합니다.

### 12.2 성능 관련

- 원본 실적 데이터가 2만 건을 초과하면 비동기 청크 처리로 자동 전환됩니다.
- 성능 핫패스는 `for` 루프와 `Float64Array`를 사용합니다 (`forEach`/`map` 대신).
- 2단계 캐시 시스템으로 반복 렌더링 시 재계산을 방지합니다.

### 12.3 CDN 의존성

- Chart.js와 docx 라이브러리는 CDN에서 로드됩니다.
- 완전한 폐쇄망 운영 시 해당 JS 파일을 로컬에 배치하고 `index.html`의 경로를 수정해야 합니다.

### 12.4 알려진 제한

- `tab-allocation` (공장배분) 섹션은 HTML에 존재하지만 전용 탭 버튼이 없습니다. `등급별현황/수입관리` 탭에서 함께 렌더링됩니다.
- 수입 선적 데이터(`getImportShipmentRows`)는 데이터 번들이 아닌 하드코딩된 값을 사용합니다.
- 푸터의 저작권 표시에 오타가 있습니다 ("Copywright" → "Copyright").

### 12.5 인쇄

`@media print` 규칙으로 헤더, 탭 네비게이션, 툴바, 페이지 노트, 푸터가 자동 숨김 처리됩니다.
