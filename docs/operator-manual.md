# 운영자 매뉴얼

## 1. 목적과 범위
이 문서는 대시보드 운영 담당자가 계정, 공지, 일정, 거래처, 원본 실적 데이터를 관리할 때 필요한 절차와 주의사항을 정리한 문서입니다.

## 2. 시스템 개요
본 시스템은 정적 웹 페이지 기반 대시보드입니다. 주요 구성은 다음과 같습니다.

| 구성 요소 | 역할 |
| --- | --- |
| `login.html` | 로그인 화면 |
| `index.html` | 메인 대시보드 |
| `js/app.js` | 차트, 연도 전환, 데이터 입력, 보고서 내보내기, 챗봇 |
| `js/admin-features.js` | 공지사항, 일정, 사용자 관리 |
| `js/storage.js` | 브라우저 저장소 추상화 |
| `js/dashboard-data.js` | 기본 대시보드 데이터 |

## 3. 실행과 접속
로컬 테스트 또는 사내 파일 서버 환경에서 정적 파일로 실행할 수 있습니다.

1. 프로젝트 루트에서 웹 서버를 실행합니다.

```bash
python -m http.server 8787 --bind 127.0.0.1
```

2. 브라우저에서 `http://127.0.0.1:8787/login.html`에 접속합니다.
3. 기본 계정으로 로그인합니다.

기본 계정은 다음과 같습니다.

| 아이디 | 비밀번호 | 권한 | 상태 |
| --- | --- | --- | --- |
| `dongkuk1` | `1234` | 관리자 | 활성 |
| `dongkuk2` | `1234` | 일반 사용자 | 활성 |

## 4. 저장 구조와 운영 특성

### 4.1 저장 위치
시스템 데이터는 서버 DB가 아니라 브라우저 저장소에 저장됩니다.

- 기본 저장소: IndexedDB `dongkuk_dashboard`
- 저장소 이름: `kv`
- 대체 저장소: 브라우저가 IndexedDB를 사용할 수 없으면 `localStorage`

### 4.2 주요 저장 키
운영 중 변경되는 데이터는 아래 키로 저장됩니다.

| 키 | 내용 |
| --- | --- |
| `planClipboardDataByYear` | 연도별 계획/실적 붙여넣기 데이터 |
| `rawTransactionDataByYear` | 연도별 원본 실적 데이터 |
| `gradeMacroMappings` | 검수 등급 매핑 |
| `supplierAdminItems` | 거래처 관리 데이터 |
| `noticesData` | 공지사항 데이터 |
| `schedulesData` | 팀 일정 데이터 |
| `usersData` | 사용자 데이터 |

### 4.3 운영 시 반드시 알아둘 점
- 브라우저 로컬 저장 방식이므로 PC를 바꾸거나 브라우저 프로필이 다르면 데이터가 공유되지 않습니다.
- 브라우저 사이트 데이터 삭제 시 운영 데이터가 사라질 수 있습니다.
- 서버 동기화, 중앙 백업, 권한 분리 기능은 현재 구현되어 있지 않습니다.

## 5. 계정과 권한 운영

### 5.1 사용자 등록
1. `사용자관리` 탭으로 이동합니다.
2. `+ 사용자 등록`을 클릭합니다.
3. 아이디, 비밀번호, 이름, 부서, 직급, 권한, 상태를 입력합니다.
4. `저장`을 누릅니다.

입력 규칙은 다음과 같습니다.

- 아이디: 영문, 숫자, 밑줄만 허용
- 비밀번호: 신규 등록 시 필수
- 이름, 부서: 필수
- 이메일: 입력 시 형식 검증

### 5.2 사용자 수정
- 기존 사용자 수정 시 아이디는 변경할 수 없습니다.
- 비밀번호를 비워 두면 기존 비밀번호가 유지됩니다.
- 상태를 `inactive`로 변경하면 해당 계정은 로그인할 수 없습니다.

### 5.3 사용자 삭제
- 현재 로그인한 계정은 삭제할 수 없습니다.
- 삭제 전 확인창이 표시됩니다.

### 5.4 권한 운영 주의
현재 `role` 값은 사용자 정보와 목록 표시에 사용되지만, 메뉴 접근 자체를 제한하지는 않습니다. 즉, 로그인한 사용자는 현재 UI 기준으로 관리 탭을 모두 볼 수 있습니다. 운영 시에는 계정 발급 정책으로 접근 대상을 통제해야 합니다.

## 6. 정기 운영 절차

### 6.1 월별 계획/실적 데이터 갱신
대상 화면: `부재료실적 모니터링 > 월별 계획 대비 실적 데이터`

1. 입력 패널을 펼칩니다.
2. 엑셀에서 월 헤더와 인천/포항의 계획, 실적 4개 행을 복사합니다.
3. 그리드에 붙여넣습니다.
4. `붙여넣은 값 적용`을 클릭합니다.
5. KPI, 계획 차트, 누계 상세 표가 갱신되었는지 확인합니다.

복원 방법:
- `기본 데이터 복원`을 누르면 해당 연도의 기본 수급계획 데이터로 되돌립니다.

### 6.2 원본 실적 데이터 갱신
대상 화면: `부재료실적 모니터링 > 거래처 납품 실적관리 데이터 (인천공장)`

1. 대상 연도를 먼저 선택합니다.
2. 입력 패널을 펼칩니다.
3. 엑셀에서 A:E 열을 다음 순서로 복사합니다.
   `전기일 / 공급업체 / 검수등급 / 단가 / 입고금액`
4. 그리드에 붙여넣습니다.
5. `원본 데이터 반영`을 클릭합니다.
6. 거래처 추이, 구매실적, 등급별현황/수입관리 화면이 정상 갱신되었는지 확인합니다.

운영 메모:
- 원본 데이터는 연도별로 따로 저장됩니다.
- 2만 건을 초과하면 비동기 처리로 진행률 안내가 표시됩니다.
- `선택 연도 초기화`를 누르면 해당 연도의 원본 실적 데이터를 제거합니다.

주의:
- 화면 그리드의 4열 제목은 `입고량`으로 보이지만, 실제 붙여넣기 파서는 4열을 `단가`로 해석하고 `입고금액 / 단가`로 수량을 계산합니다. 원본 입력 시에는 반드시 `단가`를 넣어야 합니다.

### 6.3 검수 등급 매핑 관리
대상 화면: `부재료실적 모니터링 > 검수 등급 매핑 관리`

1. 상위 분류를 선택합니다.
2. 상세 검수등급명을 입력합니다.
3. `매핑 추가`를 누릅니다.
4. 필요 시 각 매핑 칩의 `삭제` 버튼으로 개별 삭제합니다.
5. 전체를 원복하려면 `기본값 복원`을 사용합니다.

영향 범위:
- 원본 실적 데이터를 기반으로 한 등급별 현황, 배분 현황, 비교 표에 바로 반영됩니다.

### 6.4 거래처 관리
대상 화면: `거래처 관리`

등록/수정 필수 항목:
- 코드
- 거래처명
- 지역
- 대표자
- 연락처

추가 관리 항목:
- 신뢰등급
- 월 공급능력(톤)
- 금년 납품량(톤)

운영 포인트:
- 신규 등록 시 거래처 코드는 자동 제안됩니다.
- 수정 모드에서는 코드가 읽기 전용입니다.
- 삭제 시 확인창이 표시됩니다.

### 6.5 공지사항 운영
대상 화면: `공지사항`

등록 절차:
1. `+ 등록` 클릭
2. 제목, 내용, 작성자, 비밀번호 입력
3. 필요 시 `필독` 체크
4. `저장`

수정/삭제 규칙:
- 수정과 삭제 모두 게시글 비밀번호가 필요합니다.
- 비밀번호가 일치하지 않으면 수정/삭제할 수 없습니다.

운영 주의:
- 기본 제공 공지 2건은 시스템 기본값으로 포함됩니다.
- 기본 공지를 화면에서 삭제해도 새로고침 후 다시 나타날 수 있습니다.
- 지속적으로 유지하려는 공지는 신규 공지로 별도 등록하는 방식이 안전합니다.

### 6.6 팀 일정 운영
대상 화면: `공지사항 > 팀 일정관리`

지원 일정 유형:
- 오전반차
- 오후반차
- 반반차1
- 반반차2
- 반반차3
- 반반차4
- 휴가
- 교육
- 출장
- 외근

운영 규칙:
- `휴가`, `교육`, `출장`은 종료일 입력이 가능합니다.
- 종료일이 시작일보다 빠르면 저장되지 않습니다.
- 날짜를 클릭하면 해당 날짜 일정 상세와 삭제 버튼이 표시됩니다.

### 6.7 보고서 내보내기
헤더의 `보고서 내보내기`는 현재 선택 연도 기준 DOCX를 생성합니다.

포함 범위:
- `부재료실적 모니터링`
- `등급별현황/수입관리`

비포함 범위:
- 거래처 관리
- 공지사항
- 팀 일정관리
- 사용자관리

## 7. 백업과 복구 가이드

### 7.1 백업 권장 방식
현재 시스템에는 전용 백업 기능이 없습니다. 운영 데이터 보존을 위해 다음을 권장합니다.

- 원본 엑셀 파일을 별도 보관
- 주기적으로 브라우저 저장소 값을 백업
- 운영에 사용하는 브라우저와 프로필을 고정

### 7.2 장애 시 확인 순서
1. 브라우저를 새로고침합니다.
2. 조회 연도와 로그인 계정을 확인합니다.
3. 브라우저 개발자도구에서 IndexedDB 또는 localStorage 데이터 유무를 확인합니다.
4. 필요 시 원본 엑셀에서 데이터를 다시 붙여넣습니다.

## 8. 운영 유의사항
- 계정 비밀번호와 공지 비밀번호는 현재 브라우저 저장소에 보관됩니다.
- 서버 측 인증과 암호화 저장은 구현되어 있지 않습니다.
- 정식 운영 환경으로 확대하려면 서버 DB, 중앙 인증, 권한 제어를 별도로 설계해야 합니다.
